# Really Get Books In Order Full Review And Fix

Status: completed.

## Problem Statement

The user bookcase can still show `Your bookcase is empty` after adding a book, and books can appear to disappear after refresh. This must be treated as a data-flow correctness bug, not a visual glitch.

The fix must make the Books feature boring and reliable:

- Create a book.
- See it immediately.
- Refresh the browser.
- Still see it.
- Log out and back in.
- Still see the same user-owned books.
- Run against the intended local app instance.
- Pass headed browser tests that prove the behavior in the real UI.

## Added Requirement: Single Local Runtime And Docker Cleanup

The local machine must not have multiple app runtimes serving different builds while this bug is being diagnosed.

Target:

- Run exactly one web app instance during verification.
- Prefer local `dotnet run` for the app under test so source edits are immediately represented.
- Keep Docker only for backing services during local debugging:
  - Postgres
  - Redis
  - Seq
- Stop and remove stale web/app containers and exited junk containers that can confuse ports or logs.
- Do not remove Docker volumes unless explicitly requested; the bug is about correctness and should be proven without silently deleting evidence.
- Re-check listening ports before and after cleanup.

Acceptance:

- `http://127.0.0.1:5099` is the only app URL under test.
- Docker app port `7186` is not serving a stale web container during verification.
- Postgres, Redis, and Seq are the only expected Docker containers left running.

## Added Requirement: CRUD Correctness And Performance

The bookcase must avoid wasteful and fragile data flow:

- No duplicate UI fetch loops after save.
- No stale empty read is allowed to overwrite freshly saved state.
- Create, read, update, and delete must be verified through the browser and through the API/database path.
- User-owned book rendering should be bounded and predictable:
  - one strong-consistency load when the page needs user books,
  - local state update after successful mutations,
  - explicit reload only when needed after auth/page transitions,
  - no repeated reload loop from component lifecycle churn.

Acceptance:

- Adding a book updates the user bookcase immediately.
- Refreshing the browser preserves the saved book.
- Editing updates the visible shelf/table without a duplicate stale reload.
- Deleting removes the book and it stays gone after refresh.
- Logs/tests show the API returns the persisted row for the same authenticated user.

## Added Requirement: Integer Book Ids For Navigation

Persisted user books must use integer ids generated by Postgres and those ids should be the canonical navigation identity.

Current finding:

- `Book.Id` is already an `int`.
- Existing migrations/model snapshot show `UseIdentityByDefaultColumn` for `Book.Id`, so Postgres is already auto-generating ids.
- `BookEntityTypeConfiguration` currently only says `HasKey(m => m.Id)` and relies on EF/Npgsql conventions for identity generation. The fix should make this explicit with `ValueGeneratedOnAdd()` and verify the migration/model snapshot stays correct.
- User-book links already carry integer ids, but the modal query shape is noisy: `bookSource=user&bookId=123&bookMode=view`.
- Author bookcase entries are not Postgres rows. They currently use negative in-memory ids plus slug-based links, for example `bookSource=author&bookId=ship&bookMode=view`. This overloads `bookId` as both a user integer id and an author slug.

Target:

- User-owned persisted books use only positive Postgres-generated integer ids for API calls, UI links, modal state, cleanup, and tests.
- Author catalog books must not pretend to be Postgres rows. Either:
  - keep them as a separate static catalog with a clearly named `authorBookId` integer query parameter, or
  - keep slugs but rename the query parameter to `authorSlug` so it can never be confused with a persisted book id.
- Preferred cleanup: `bookId` means persisted user book integer id only. Author links use `authorBookId`.

## Investigation Findings So Far

- Both `http://127.0.0.1:5099` and Docker `https://localhost:7186` are currently listening locally. This creates a real risk of testing a stale container or a different app instance than the one just built.
- The UI currently mixes direct child component state updates (`@ref` into `UserBookcase`) with route navigation (`Nav.NavigateTo("/books")`) and async reloads. That is fragile in Interactive Auto because prerender, Server interactivity, and WebAssembly interactivity can run different service implementations.
- `UserBookcase` catches all load exceptions and only shows a generic error. The current bug needs richer temporary diagnostics while fixing.
- The earlier cache-bypass API option could skip `HybridCache`, but the UI could still race:
  - save applies the saved book to local component state,
  - navigation closes the modal,
  - another lifecycle load can replace local state with an empty list if auth/user/context is not what we think it is.
- E2E currently proves one create/edit flow in a clean test user, but it does not yet prove repeated refreshes, local seeded users, or the exact failure pattern the user sees.

## Final Root Cause

The first failure was a real local-runtime issue plus a real UI-state bug:

- The machine had two app runtimes serving different builds: a local process on `127.0.0.1:5099` and a stale Docker web container on `127.0.0.1:7186`.
- User books were saved successfully through the API, but `UserBookcase` decided whether to load by requiring a `ClaimTypes.NameIdentifier` claim from the component authentication state.
- In the failing Interactive Auto path, the UI was authenticated but the component auth state could expose only a name fallback at that point. Because the user key was treated as empty, the bookcase never called `/api/books` and rendered `Your bookcase is empty`.
- The old `@ref` flow made this worse: the modal applied local state, then route navigation and lifecycle reloads could replace it.
- Mobile E2E also caught a static-render edge: the editor close control was a Blazor-only button. It is now a normal link, so close works before hydration too.

Final shape:

- Docker now only runs backing services during the local validation session: Postgres, Redis, Seq.
- `http://127.0.0.1:5099` is the one app URL under test.
- `bookId` now means persisted user book integer id only.
- Author catalog books use `authorBookId`.
- User bookcase state is owned by a scoped Books slice state object, not parent component `@ref` calls.
- The UI loads user books for any authenticated component principal, while server-side ownership resolves through Identity and remains enforced by `OwnerUserId`.
- Browser E2E now proves create, refresh, edit, refresh, delete, and refresh for both a newly registered user and the seeded local user.

## Non-Goals

- Do not touch deployment.
- Do not change the book visual design unless necessary for test selectors or state clarity.
- Do not add uploads, images, or file storage.
- Do not remove authentication. User books remain per-user.
- Do not hide the bug by only patching the UI. Persistence and reload behavior must be proven.
